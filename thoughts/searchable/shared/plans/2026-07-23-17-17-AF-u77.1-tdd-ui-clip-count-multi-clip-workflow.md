# AF-u77.1 UI Multi-Clip YouTube-to-Reels — Revised TDD Implementation Plan

## Overview

Let a reel-af-ui user select the A1 preset, provide either a YouTube URL or an
org-owned uploaded file, choose `clip_count` from 1 through 8, and receive one
progressively rendered result card per planned clip.

The planner already produces ordered multi-clip hook plans, and the publisher already
rewrites clip composite refs for object storage. This slice connects those capabilities
through explicit, testable boundaries:

1. the web submit boundary validates and forwards `clip_count`;
2. publication returns a required, typed `clip_render_descriptors` manifest derived
   from the exact hook plan it publishes;
3. the owned poll endpoint preserves that manifest;
4. the browser loads one explicitly served dependency-free module;
5. that module projects URL or FILE render inputs, runs render jobs serially, retains
   partial failures, and reports progressive snapshots;
6. the page renders mutually exclusive singular and A1 result surfaces.

This revision resolves every critical and warning-level finding in the linked review.
It deliberately changes the original plan where the real code owns data or promises
differently than the first draft assumed.

## Current State and Verified Boundaries

### Web submission

- `TRANSCRIPT_ALLOWED_INPUT_KEYS` currently permits only `source_url`, `source`, and
  metadata, so client `clip_count` is rejected
  (`web/reel_jobs.py:136-139`;
  `tests/web/test_a1_transcript_submit.py:101-122`).
- `_parse_composite_count()` is the existing bounded integer pattern, but its public
  error code is `invalid_count`; A1 will use its own parser because this plan freezes
  `invalid_clip_count` as a distinct API contract
  (`web/reel_jobs.py:351-373`).
- The transcript URL and FILE branches currently return different source shapes and
  `_resolve_cp_input()` converts an owned FILE handle to a fresh `source_url` before
  CP dispatch (`web/reel_jobs.py:601-638`; `web/server.py:387-411`).
- Authentication, authorization, canonicalization, presigning, row insertion, and CP
  dispatch already occur in a fail-closed order (`web/server.py:469-512`).

### Planner, publication, and result ownership

- `transcript_to_plan(..., clip_count=1)` validates count before transcription and
  delegates multi-clip production to the existing planner
  (`src/reel_af/app.py:1858-1917`).
- Planner output contains top-level refs and `clip_count`, but not a parsed hook plan or
  a hosted per-clip map (`src/reel_af/planner/pipeline.py:584-594`).
- `publish_a1_artifacts()` is the only component that owns both the parsed hook plan
  and the raw-to-hosted `composite_urls_by_ref` map
  (`src/reel_af/storage.py:201-287`).
- `transcript_to_plan()` replaces its raw planner result with the sync or async writer
  result; app-layer code cannot recover hosted clip 2+ refs without violating that
  boundary (`src/reel_af/app.py:1897-1915`).
- `clip_dispatches` already means a post-dispatch execution summary
  `{idx, idempotency_key, execution_id}`. This plan does not reuse that name
  (`src/reel_af/planner/dispatch.py:98-125`).

### Browser and HTTP delivery

- The current A1 chain submits leg 2 once with hard-coded clip index 1 and request key
  `${clientRequestId}:render` (`web/index.html:1231-1256`).
- A1 source input is an exactly-one union: `{source}` for FILE or `{source_url}` for
  URL. Spreading it after adding `clip_count` would leak a leg-1-only field into leg 2
  (`web/index.html:1162-1169,1245-1252`).
- The page is a classic script with an async IIFE, and `CFG` is lexical to that IIFE
  (`web/index.html:667-678`).
- Flask disables general static serving with `static_folder=None`; a new `.mjs` needs
  an explicit named route (`web/server.py:1216-1266`).
- `execute()` has a bounded submit-retry window, and `awaitResult()` has a per-execution
  poll deadline. Neither cancels accepted CP work or aborts an in-flight fetch
  (`web/index.html:1268-1295,1556-1587`).
- `finish()` owns the current singular result lifecycle: completion stage, visibility,
  download-link discipline, and completion logging
  (`web/index.html:1604-1617`).
- CI currently runs Ruff and Pytest only (`.github/workflows/ci.yml:19-31`).

## Desired End State

### Observable Behaviors

1. Given an authenticated transcript submission with absent or valid `clip_count`,
   the server normalizes it to an integer in `1..8` and forwards it in URL and FILE
   CP bodies and normalized `params`.
2. Given an invalid value, the server returns `400 invalid_clip_count` before
   presigning, row creation, or CP dispatch, with pinned mixed-error precedence.
3. Given a successful planner result, publication returns
   `clip_render_descriptors` for every clip, including exactly one descriptor when
   `clip_count=1`.
4. Given bucket publication, every descriptor contains its hosted per-clip composite
   URL; given no-bucket local development, the same required shape contains local refs
   and remains explicitly non-browser-deliverable.
5. Given a custom artifact writer, its final output must satisfy the same publication
   contract or the reasoner returns `dsl_artifact_unavailable`.
6. Given an owned transcript execution, polling preserves the ordered descriptor array
   byte-for-byte.
7. Given the A1 page, `/a1-render-plan.mjs` loads with `text/javascript`, and the
   production `rollA1()` function invokes the same exported helpers tested by Node.
8. Given URL or FILE mode, each render input contains exactly one source arm and never
   contains `clip_count`.
9. Given N descriptors, the browser dispatches and polls exactly N renders in order
   with maximum in-flight render count 1 and shows each terminal outcome before the
   next clip finishes.
10. Given mixed render outcomes, successful downloads and failed cards remain visible;
    post-dispatch failures retain `execution_id` and failure phase.
11. Given any new Roll or idle preset switch, stale singular and A1 results are
    cleared, and only the owning result surface is visible.

## Locked Contracts and Decisions

### Transcript count

```text
TRANSCRIPT_CLIP_COUNT_DEFAULT = 1
TRANSCRIPT_CLIP_COUNT_MIN     = 1
TRANSCRIPT_CLIP_COUNT_MAX     = 8
public invalid code          = invalid_clip_count
```

Accept an integer or whitespace-padded decimal digit string. Reject `bool`, float,
non-digit strings, zero, negative values, and values above 8.

Inside the `TARGET_TRANSCRIPT` branch, perform these operations exactly once and in
this order:

```python
_reject_unsupported_fields(raw_input, TRANSCRIPT_ALLOWED_INPUT_KEYS)
_reject_both_sources(raw_input)
clip_count = _parse_transcript_clip_count(raw_input)
# then validate the selected FILE handle or URL
```

Externally visible precedence is therefore:

1. existing request/body/identity/target gates;
2. unsupported transcript input field;
3. mutually exclusive source violation;
4. `invalid_clip_count`;
5. missing or malformed selected source.

Do not extract a generic bounded-integer helper in this slice. Composite `count` and
transcript `clip_count` intentionally expose different error codes.

### Publication DTO

`clip_render_descriptors` is required on every successful `transcript_to_plan()` result:

```python
class ClipRenderDescriptor(TypedDict):
    idx: int
    composite_ref: str
    idempotency_key: str
```

Freeze all of these invariants:

- `clip_count` is an integer, not a boolean, and is at least 1;
- descriptor length equals `clip_count`;
- list order and indices are exactly `1..clip_count`;
- each descriptor has exactly the three fields above;
- refs and keys are non-empty strings and unique within the list;
- descriptor 1's `composite_ref` equals the legacy top-level `composite_ref`;
- descriptor `idempotency_key` is copied byte-for-byte from the hook plan;
- bucket mode returns hosted HTTP(S) descriptor refs and no local path in the result;
- no-bucket mode returns the same shape with local refs, does not construct an S3
  client, and is explicitly a co-located-development result;
- malformed or partial publication never returns a successful manifest.

The descriptor key is planner-authored lineage/debug identity. It is not used as an
HTTP `Idempotency-Key`.

### URL and FILE source union

Leg 1 browser input is:

```text
{source: upload_handle, clip_count: N}
or
{source_url: public_url, clip_count: N}
```

Leg 2 browser input is:

```text
{source: upload_handle, composite_ref, words_ref, hook_ref, clip_idx}
or
{source_url: public_url, composite_ref, words_ref, hook_ref, clip_idx}
```

Leg 2 never includes `clip_count`, descriptor `idempotency_key`, or `target` under
`input`. FILE mode reuses the captured upload handle; the existing server presigns it
freshly for each CP dispatch.

### Idempotency roles

- Leg 1 HTTP identity is the per-Roll `clientRequestId`.
- Leg 2 HTTP identity preserves the current one-clip request key:

  ```javascript
  descriptorCount === 1
    ? `${clientRequestId}:render`
    : `${clientRequestId}:render:${descriptor.idx}`
  ```

- For N>1, keys are distinct and retries for a given clip reuse that clip's key.
- `clip_render_descriptors[].idempotency_key` remains lineage-only and is exposed on
  render-plan items as `lineageIdempotencyKey`; it is never sent as the HTTP header or
  under CP `input`.

### Render concurrency and progress

`A1_RENDER_MAX_IN_FLIGHT` is an exported module constant with value 1 and is the
single source of truth for the resource policy. The runner uses a serial `for...of`
loop, not `Promise.all` or `Promise.allSettled`. After each clip settles it invokes
`onOutcome(outcomes.slice())` before dispatching the next clip. Node tests import the
constant, assert it equals 1, and measure peak active work independently.

Outcome shapes are:

```javascript
// success
{ idx, status: "succeeded", phase: "complete", execution_id, result }

// execute failed before an execution id was observed
{ idx, status: "failed", phase: "dispatch", execution_id: null, error }

// poll/render failed after dispatch
{ idx, status: "failed", phase: "poll", execution_id, error }
```

Use named application constants for `succeeded`, `failed`, `complete`, `dispatch`, and
`poll`. Do not rename CP status strings or JavaScript platform values.

### Timeouts and cancellation

- Leg 1 and every render submit independently use the existing
  `CFG.api.submitPendingTimeoutMs`.
- Leg 1 and every accepted render independently use the existing `CFG.api.timeoutMs`
  polling deadline.
- Serial N-clip execution may consume N render poll windows; there is no additional
  batch deadline.
- Existing deadlines are checked between fetches; this plan does not promise
  `AbortController` cancellation of an in-flight request.
- Navigation, tab closure, a client timeout, or DOM-rendering failure does not cancel
  already accepted CP work.
- No cancellation route is added; the existing
  `/api/v1/executions/{id}/cancel` 404 contract remains unchanged
  (`tests/web/test_routes.py:22-27`).
- Each reasoner retains ownership of its work directory and cleanup behavior. Serial
  dispatch avoids adding overlapping heavyweight render work.

## Workflow Closure

```text
authenticated browser
  -> transcript submit {URL|FILE, clip_count:N}
  -> web validates once, canonicalizes, presigns FILE, dispatches leg 1
  -> transcript_to_plan plans N clips
  -> publish_a1_artifacts rewrites refs and authors clip_render_descriptors[1..N]
  -> transcript_to_plan validates final writer output
  -> CP result
  -> owned poll preserves descriptors
  -> page dynamically imports /a1-render-plan.mjs
  -> page projects exactly one render source arm
  -> for descriptor 1..N, serially:
       submit render with explicit target/action idempotency key
       retain execution id
       poll terminal result
       append outcome
       project/render progressive cards
  -> render final snapshot, advance done, log success/failure counts
```
