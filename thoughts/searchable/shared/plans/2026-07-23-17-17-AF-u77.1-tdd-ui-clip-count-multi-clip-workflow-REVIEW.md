# Plan Review Report: AF-u77.1 UI Multi-Clip YouTube-to-Reels

## Review Summary

The plan is directionally strong but is **not ready for implementation**. The existing
single-clip web path, multi-clip planner, publication layer, and render path are all
real and have clean focused tests. The blockers are in the proposed joins between
those layers:

1. `transcript_to_plan()` cannot construct post-publication per-clip hosted refs from
   the values it owns.
2. The proposed browser module has no production HTTP route/import contract and reads
   a page-local `CFG` that is unavailable in ES-module and Node scope.
3. The new render-input projection drops the existing FILE-mode source handle.
4. The default one-clip result may omit the array that the browser requires.
5. The blocking JavaScript test is not wired into CI, uses a non-matching glob, and
   does not prove that production `rollA1()` calls the tested helper.
6. Partial-failure rendering, result-surface lifecycle, and render concurrency remain
   materially under-specified.

## Contract Review

### Well-Defined

- ✅ Invalid web inputs are rejected before row creation and CP dispatch:
  `web/server.py:469-492`.
- ✅ Planner `clip_count` validation runs before transcription, planning, or
  publication: `src/reel_af/app.py:1876-1886`.
- ✅ The renderer remains one-video-per-dispatch and returns a browser-deliverable
  `download_url`: `src/reel_af/app.py:1672-1735,1822-1855`.
- ✅ Publication already uploads distinct per-clip composites and preserves planner
  idempotency keys: `src/reel_af/storage.py:225-287`;
  `tests/planner/test_reasoner.py:298-339`.
- ✅ The authenticated poll boundary can carry additive result fields without a new
  endpoint version: `web/server.py:355-379,1013-1039`.

### Missing or Unclear

- ❌ **Publication ownership is wrong in Behavior 4.** The plan assumes an
  `hook_plan_dict` exists after publication (`reviewed plan:581-598`), but
  `transcript_to_plan()` owns only the planner result and then the writer result
  (`src/reel_af/app.py:1897-1915`). The only post-publication per-clip URL map is the
  private `composite_urls_by_ref` in `publish_a1_artifacts()`
  (`src/reel_af/storage.py:220-287`).
- ❌ **Default cardinality is contradictory.** Behavior 4 permits omitting
  `clip_dispatches` for `clip_count=1` (`reviewed plan:535-539`), while Behavior 6
  unconditionally calls `.map()` on it (`reviewed plan:765-788`).
- ❌ **FILE-mode source ownership is lost.** Current leg 2 preserves either
  `{source}` or `{source_url}` (`web/index.html:1162-1169,1245-1252`). The proposed
  helper accepts only `source_url`, but the planner result does not return one
  (`src/reel_af/planner/pipeline.py:584-594`).
- ❌ **The HTTP error contract conflicts.** Desired behavior promises
  `invalid_clip_count` (`reviewed plan:138-142`); the tests and Green sketch require
  `invalid_count` (`reviewed plan:333-380`).
- ⚠️ **Idempotency roles are not reconciled.** Planner keys are immutable
  content/lineage identifiers (`src/reel_af/planner/serialize.py:504-515`), while the
  browser creates UI-action keys (`reviewed plan:765-769`). The plan must explicitly
  state which value is the HTTP idempotency header and which is lineage-only.
- ⚠️ **No focused poll contract test names the new field.** Generic passthrough exists,
  but a succeeded CP body containing the exact multi-clip descriptor array should be
  pinned through `GET /api/v1/executions/{id}`.

### Recommendations

- Make publication the owner of the compact post-publication descriptor/manifest.
- Require a non-empty descriptor array on every successful result, including one
  entry for the default case.
- Define leg-2 source as the exactly-one union `{source}` or `{source_url}` and exclude
  the leg-1-only `clip_count`.
- Choose one web error code and use it in prose, implementation, tests, and docs.
- Document planner lineage identity separately from HTTP request idempotency.

## Interface Review

### Well-Defined

- ✅ Existing target names and reasoner signatures are correct:
  `web/reel_jobs.py:23-35,72-85`; `src/reel_af/app.py:1858-1869`.
- ✅ The proposed normalized integer is compatible with the reasoner.
- ✅ Existing browser adapter signatures can support dependency injection:
  `execute(target, input, clientRequestId)` and `awaitResult(id)` at
  `web/index.html:1268-1295,1556-1587`.

### Missing or Unclear

- ❌ **The new module cannot be served.** Flask uses `static_folder=None` and exposes
  only named page/config routes (`web/server.py:1216-1266`). The plan creates
  `web/a1-render-plan.mjs` but does not touch `web/server.py`.
- ❌ **The import/binding mechanism is unspecified.** The page is a classic inline
  script with an async IIFE (`web/index.html:667-678`). A separate
  `<script type="module">` does not put named exports into that IIFE's lexical scope.
- ❌ **The module has a hidden global.** The closure prose promises plan items contain
  `target`, but the Green sketch omits it and reads `CFG.a1.renderTarget`
  (`reviewed plan:765-778`). `CFG` is inaccessible from the new module and Node.
- ❌ **The source interface is not a union.** FILE mode needs `{source: handle}`;
  URL mode needs `{source_url: URL}`. The proposed destructuring accepts only the
  latter and a naive `...sourceInput` fix would leak disallowed `clip_count` into leg 2.
- ❌ **The publication interface cannot return the requested hosted descriptor shape**
  without a storage change or a refetch that the plan forbids.
- ❌ **Result-surface ownership is incomplete.** `finish()` currently advances to
  `done`, shows `#result`, manages the download link, and logs completion
  (`web/index.html:1604-1617`). The replacement only specifies clone/fill/append.
- ⚠️ **Names remain implementation-time placeholders.** Exact module filenames, DOM
  ids, and test globs disagree or remain TBD (`reviewed plan:200-204,655-657,706-716`).

### Recommendations

- Add an explicit `GET /a1-render-plan.mjs` route and test status/body/MIME type, or
  choose a different single-source test seam.
- Pin one binding strategy, preferably a static/dynamic import inside module scope.
- Put `target` on each render plan item and use `execute(item.target, ...)`.
- Pass an explicit leg-2 source union into `planClipRenders`.
- Define A1 versus singular result visibility, clearing, completion-stage, and
  preset-switch behavior.

## Promise Review

### Well-Defined

- ✅ Leg 1 is awaited and checked before any render dispatch:
  `web/index.html:1231-1244`.
- ✅ Submit retry reuses a bounded idempotency key, and server replay avoids a second
  CP call: `web/index.html:1268-1295`; `web/server.py:469-512`.
- ✅ `Promise.allSettled` is the correct primitive for preserving sibling outcomes
  once concurrency and UI-consumption contracts are fixed.
- ✅ Polling already treats 429/5xx as transient and enforces terminal delivery rules:
  `web/server.py:1017-1039`.

### Missing or Unclear

- ❌ **The tested helper is not the production seam.** The Node test explicitly does
  not call `rollA1()` (`reviewed plan:258-296`). It can pass while production retains
  the current single-dispatch implementation.
- ❌ **The JavaScript test is not in CI.** Current CI runs Ruff and Pytest only
  (`.github/workflows/ci.yml:25-31`); `.github/workflows/ci.yml` is absent from the
  plan's touched files.
- ❌ **The declared test glob does not select the planned test.**
  `web/a1-render-plan.test.mjs` does not match `web/*.results.test.mjs`
  (`reviewed plan:200-212,711-716`).
- ❌ **Concurrency is an unreviewed behavior change.**
  `Promise.allSettled(plan.map(...))` starts up to eight heavyweight render jobs at
  once, while the existing CLI dispatches and polls serially
  (`scripts/ingest_source.py:289-309`).
- ❌ **The partial-failure "integration" test never invokes result rendering.**
  It only asserts that outcome objects contain `idx/status`
  (`reviewed plan:903-918`).
- ⚠️ **Timeout/cancellation semantics are incomplete.** Browser polling has a local
  deadline but no fetch abort and accepted CP work is not cancelled.
- ⚠️ **A batch barrier hides completed clips.** No result is shown until the slowest
  sibling settles or times out.

### Recommendations

- Add an exact production-wiring assertion that `rollA1()` imports and invokes the
  tested helper.
- Add a pinned Node setup/test step to CI using the exact filename.
- Default to serial render dispatch to match current behavior, or define/test a named
  maximum in-flight count with deployment-capacity evidence.
- Extract a pure result view-model projection used by the DOM renderer and test mixed
  success/failure cardinality, ordering, link discipline, and error visibility.
- State per-request, per-clip, and whole-workflow timeout behavior and that client
  timeout/navigation does not cancel accepted work.

## Data Model Review

### Well-Defined

- ✅ Hook-plan v1 has an explicit schema version and ordered clip array:
  `src/reel_af/planner/serialize.py:192-237`.
- ✅ Existing dispatch input models are typed and validated:
  `src/reel_af/planner/dispatch.py:14-27,57-95`.
- ✅ Publication is copy-on-write and removes producer-local sidecar paths:
  `src/reel_af/storage.py:215-287`.
- ✅ Adding normalized `clip_count` to JSON-backed `params` and `cp_input` requires no
  relational migration.

### Missing or Unclear

- ❌ **The descriptor is authored at the wrong boundary.** Only storage owns the
  local-to-hosted ref mapping.
- ❌ **The source model drops one arm of the existing union.**
- ❌ **The descriptor is optional where the consumer requires it.**
- ⚠️ **`clip_dispatches` already names a different schema.**
  `dispatch_dsl_hook_clips()` returns post-dispatch
  `{idx,idempotency_key,execution_id}` at `src/reel_af/planner/dispatch.py:98-125`;
  the plan proposes a pre-dispatch `{idx,composite_ref,idempotency_key}` shape.
- ⚠️ **Outcome failures lose recovery context.** If execute succeeds and poll later
  rejects, the proposed rejection mapper drops `execId` and failure phase.

### Recommendations

- Prefer a distinct, typed name such as `clip_render_descriptors` or `clip_manifest`.
- Freeze these invariants:
  `len(descriptors) == clip_count`, indices are exactly `1..clip_count`, refs and keys
  are non-empty, ordering is stable, refs are hosted in bucket mode, and clip 1 matches
  the legacy top-level `composite_ref`.
- Pin URL and FILE normalized shapes before and after `_resolve_cp_input`.
- Define outcome fields for dispatch failure versus poll/render failure and retain
  `execution_id` whenever dispatch succeeded.

## Critical Issues (Must Address Before Implementation)

1. **Publication/data ownership: hosted per-clip refs are unavailable in `app.py`.**
   - Impact: clip 2+ either uses a local path, the wrong top-level clip-1 ref, or an
     undefined hook-plan value.
   - Recommendation: evolve `publish_a1_artifacts()` to return a typed descriptor
     manifest from its rewritten in-memory hook plan; define custom-writer and no-bucket
     behavior.

2. **Browser module delivery and binding are missing.**
   - Impact: `/a1-render-plan.mjs` returns 404, and exports are not visible to the
     current classic-script IIFE.
   - Recommendation: add an explicit route and one exact import strategy with Flask
     and page-contract tests.

3. **The module's `CFG` dependency makes it fail in Node and browser module scope.**
   - Impact: no render dispatch occurs.
   - Recommendation: put `target` on plan items or inject it explicitly.

4. **The render input breaks FILE mode and risks leaking leg-1-only `clip_count`.**
   - Impact: existing DROP FILE A1 submissions fail before CP dispatch.
   - Recommendation: project exactly one source arm and test URL/FILE bodies through
     the real HTTP boundary.

5. **The default one-clip schema is optional at the producer and mandatory at the UI.**
   - Impact: the default workflow can fail on `undefined.map`.
   - Recommendation: require exactly one descriptor for `clip_count=1`.

6. **The blocking JS closure test is not executable as promised.**
   - Impact: CI can remain green with single-dispatch production behavior.
   - Recommendation: pin filenames, add Node CI, and test production wiring as well as
     helper internals.

7. **Partial-failure/result lifecycle is not closed.**
   - Impact: cards may remain hidden, stale singular results may coexist, or successful
     siblings may be dropped even when helper tests pass.
   - Recommendation: define DOM ownership/lifecycle and test a pure result projection
     consumed by the renderer.

8. **Render concurrency/resource behavior is unspecified.**
   - Impact: one click can start eight heavyweight render/source-download jobs, unlike
     the current serial precedent.
   - Recommendation: choose serial or a named tested concurrency bound.

9. **The public invalid-count error code is contradictory.**
   - Impact: implementation and tests can satisfy a different API from the Desired End
     State.
   - Recommendation: choose `invalid_count` or `invalid_clip_count` once and update
     every layer.
